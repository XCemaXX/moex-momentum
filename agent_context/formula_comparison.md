# Formula comparison — `simple` vs `curve_fit` (статистический gate)

Research-документ (task 009). Сравнение двух momentum-сигналов на полной панели
**2012-12 → 2026-04** (160 месячных доходностей, equal-weight квартили,
комиссия 0.0005 за сторону уже зашита в NAV).

- `simple` = `r(12-1) / σ(12)`
- `curve_fit` = `(0.9·r(12-1) + 0.1·r(6-1)) / σ(12)` — текущий production default

Источник чисел: `scripts/compare_formulas.py` (читает on-disk
`data/momentum/{simple,curve_fit}/q_values.csv` + `holdings/`, ничего не
пересчитывает в самой стратегии). Скрипт детерминирован (numpy seed=0).

**Замечание про stack.** scipy не входит в locked stack, поэтому:
параметрический paired t-test / Wilcoxon заменены на **sign-flip permutation
test** (10000 перестановок); Spearman ρ берётся через
`DataFrame.corr(method="spearman")` (собственная rank-реализация pandas — НЕ
scipy; `Series.corr` делегировал бы в `scipy.stats.spearmanr`); bootstrap — на
чистом numpy.

## 2. Spearman ρ (монотонность квартилей)

Конвенция: ординал квартиля Q1=1 … Q4=4; коррелируем с **средней месячной
доходностью** квартиля. ρ = **−1** ⇒ идеально монотонно убывает (Q1 лучший,
Q4 худший); ρ ≈ 0 ⇒ порядок не монотонен.

| Signal     | Spearman ρ |
|------------|-----------:|
| simple     |  **−1.0000** |
| curve_fit  |  **−1.0000** |

**Caveat:** ρ считается по 4 точкам (квартили). На n=4 Spearman грубый — для
любого монотонного порядка он ровно −1, промежуточных градаций «насколько»
монотонно почти нет. Поэтому «ρ=−1 у обоих» означает лишь «оба монотонны», а не
«одинаково робастны»; различать формулы этим критерием больше нельзя.

**Оба сигнала идеально монотонны** — и по средней месячной доходности, и по
terminal NAV:

- simple:    NAV Q1=14.46 > Q2=3.09 > Q3=2.00 > Q4=0.45
- curve_fit: NAV Q1=12.76 > Q2=3.85 > Q3=1.94 > Q4=0.42

> **Сюрприз / отмена старой посылки.** Изначальная мотивация task 009
> (`agent_context/backtest_findings.md`, Phase 9) — «curve_fit монотонный,
> simple — нет» — на текущей вселенной **больше не выполняется**. Прошлый вывод
> был сделан до фильтра `TOP_N_LIQUID=100` (см. коммит «cahnge filter
> MIN_MONTHLY_VALUE to TOP_N_LIQUID»). На новой, более ликвидной вселенной
> `simple` стал монотонным тоже. Монотонность **перестала быть
> различающим критерием** между формулами.

## 3. Статистика на разнице Q1 (simple vs curve_fit)

### Block bootstrap 95% CI на `NAV_simple / NAV_curve_fit − 1` (Q1)

Block length = 12 мес, 10000 ресэмплов, seed=0. Месяцы ресэмплятся блоками;
один и тот же набор блок-индексов применяется к обоим сигналам (pairing
сохраняется), затем каждая серия компаундится в terminal NAV.

- Point estimate: **+13.33%** (simple Q1 на ~13% выше)
- 95% CI: **[−10.29%, +42.12%]**
- **0.0 ВНУТРИ CI ⇒ разница Q1-NAV статистически НЕ значима.**

### Paired sign-flip permutation test (месячные Q1 returns, simple − curve_fit)

10000 перестановок знака, seed=0. H0: парные разницы симметричны относительно 0.

- Observed mean diff: **+0.000806 / месяц** (≈ +0.08% в месяц в пользу simple)
- Two-sided p-value: **0.2171**
- **p ≫ 0.05 ⇒ не отвергаем H0.** Месячные доходности Q1 у двух формул
  статистически неразличимы.

Обе проверки согласованы: точечно `simple` Q1 выше, но разрыв в пределах шума —
ни bootstrap CI, ни permutation-тест значимости не дают.

## 4. Вывод

- **default остаётся `curve_fit`, потому что:**
  1. Преимущество `simple` по Q1 (NAV +13.3%, CAGR +1.14 п.п., Sharpe +0.044)
     **статистически не значимо** — 0 лежит внутри bootstrap 95% CI
     [−10.29%, +42.12%], permutation p=0.2171. Менять production-default ради
     разницы в пределах шума нельзя.
  2. Старый различающий критерий — монотонность — **обнулился**: на текущей
     вселенной (`TOP_N_LIQUID=100`) обе формулы дают ρ=−1.0 и монотонный NAV.
     Монотонность больше не основание ни ЗА `curve_fit`, ни ЗА `simple`.
  3. При отсутствии значимой разницы default не трогаем по принципу инерции:
     `curve_fit` уже в проде, методология и сайт собраны под него, а смена
     влечёт Phase 12 re-verification (Jaccard к автору) без статистического
     выигрыша взамен.

- Что было бы основанием **сменить** на `simple`: значимое преимущество (CI не
  пересекает 0 / p<0.05) ИЛИ восстановление различия по монотонности в пользу
  `simple`. Сейчас нет ни того, ни другого.

- Кандидат на отдельную задачу: разница `simple`/`curve_fit` сконцентрирована в
  Q1 vs Q2 (на Q2 `curve_fit` стабильно лучше). Это сигнал, что вес r(6-1)
  перераспределяет имена между верхними квартилями, а не улучшает топ. Walk-forward
  по времени (task 009 «Не входит») покажет, устойчиво ли Q1-преимущество simple
  или это пост-2024 артефакт новой вселенной.
