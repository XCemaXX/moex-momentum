# Каталог стратегий/серий для explorer-графика (зонтик)

> Зонтичная задача: список серий, которые плагинятся в график из `task 20`, по мере роста. Каждый пункт со временем дробится в отдельную задачу. Это файл-каталог, чтобы держать видение в одном месте; реализация — итерациями.

## Идея

`task 20` строит механизм: страница `compare.html`, реестр именованных серий, пользователь сам тоглит линии. Эта задача — **что именно** в этот реестр добавлять. Каждый пункт = провайдер серий, отдающий 1-2 кривые (Q1 + бенчмарк), которые регистрируются без правок кода графика.

Гранулярность по всему каталогу: **Q1 + бенчмарки, по стратегиям** (locked в task 20). Полные Q1-Q4 — на существующих per-signal страницах, не здесь.

Оси комбинаций: `formula {simple, curve_fit} × weighting {equal, freefloat_cap, full_cap} × filter {none, mages, imoex}` (до 18 вариантов Q1). **Все комбинации pre-computed на build-time** — клиент тоглит готовое. Расчёт «на лету» в браузере отложен в `task 22` (нужен только для неограниченных пользовательских комбинаций; для фиксированного списка pre-compute достаточно).

## Заметки по пунктам

- **#4 IMOEX-filter** напрямую тестирует гипотезу #3 из `task 17`: наш юниверс шире (165 vs ~107 у автора), мы топим Q1 неликвидом. Сузить Q1 до членов IMOEX и наложить на чистый Q1 — увидеть, поднимается ли кривая. Состав IMOEX по месяцам берётся из того же analytics-эндпоинта, что и в task 19.
- **#5 cap-weighted Q1** требует cap-вектор из task 19. **Проверено (backtest.py:70-74): сейчас Q-доходность equal-weight (1/N на тикер).** Значит cap-weight — реальная альтернатива, есть с чем сравнивать. Блокер — исторический ряд числа акций (см. task 19).
- **#2 mages**: total Mages Index (как у автора, с облигациями) и Mages-Equity (только акции) — разные линии; сравнивать с Q1 только Mages-Equity (методически, см. task 2).
- **#4/#5 — это выделенные ячейки сетки** `formula×weighting×filter` (#4 = `filter=imoex`, #5 = `weighting=cap`), а не отдельные виды серий. В реестре регистрируются как обычные комбинации с соответствующим `dims`. #2/#3/#6 — серии другого рода (mages-оверлеи, индексы, прочее).
- **#6 стоп-лоссы автора** (id 135) требуют внутримесячных/дневных цен + режимных сигналов (RGBICP1Y, OECD CLI) — тяжелее серии из месячных доходностей, не сводится к weighted-sum. Отдельный провайдер, не ячейка сетки.
