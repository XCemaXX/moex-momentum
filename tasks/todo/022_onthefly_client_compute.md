# FUTURE: расчёт комбинаций стратегий «на лету» в браузере

> Backlog / exploratory. Может не понадобиться. Дефолт для explorer (`task 20/21`) — **pre-compute всех комбинаций на build-time**. Эта задача — только если возникнет потребность в *неограниченных пользовательских* комбинациях.

## Когда это вообще нужно

Explorer показывает кривые из 3 осей: `formula {simple, curve_fit} × weighting {equal, freefloat_cap, full_cap} × filter {none, mages, imoex}` = до 18 вариантов Q1.

Для **фиксированного** набора (даже 18, даже больше) — прекомпьютить все на build-time тривиально (18 × ~160 чисел), надёжно, нулевой дрейф. Клиент только тоглит готовые traces. **Это дефолт, эта задача не нужна.**

On-the-fly оправдан **только** при потребности в комбинациях, не заложенных на build-time:
- произвольные веса (свой blend free-float / full / equal),
- кастомные фильтры (свой список тикеров),
- параметрические свипы (напр. веса curve_fit 0.9/0.1 → ползунок).

## Выполнимо ли на GitHub Pages

Да, но клиентски. Pages = статика, серверного счёта нет; считает **браузер**.

Два маршрута:
1. **JS-backtest** — переписать движок на JS, отгрузить входные панели в браузер.
2. **pyodide / WASM** — гонять наш Python-движок в браузере (меньше дрейфа, тяжелее загрузка).

Что грузить в браузер: месячные доходности (~1000 тикеров × ~160 мес), cap-векторы (free-float / full) по месяцам, множества фильтров (mages / imoex) по месяцам.

## Умный компромисс (если делать)

Не считать всё в браузере. Дорогая часть — сигнал + границы квартилей — зависит от `formula × filter`, но **НЕ** от `weighting`. Поэтому:
- **членство в квартилях** (`formula × filter`) прекомпьютить на build-time (мало комбо);
- **взвешивание** применять в браузере — это взвешенная сумма доходностей по месяцам, тривиальный JS без пересчёта сигнала.

Тогда ось `weighting` (включая произвольные веса/ползунки) живая, а тяжёлая часть статична.
