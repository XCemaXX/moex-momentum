# FUTURE: Индекс магов — ингест, отображение, наложение на Q1

Не реализуется в основном плане. Анализ и детальный дизайн — отдельной итерацией.

## Что хочется

1. **Ингест индекса магов** из картинок, которые юзер вручную добавляет в репо (по типу `raw_sources/индекс_магов_*.png` → текст). Сейчас сделано руками одним claude-вызовом — нужно превратить в воспроизводимый процесс.
2. **Хранение** распарсенных составов индекса в JSONL, формат симметричен holdings (`data/mages_index/<YYYY-MM>.jsonl`):
   ```
   {"weight": 0.1653, "ticker": "SBER", "canonical": "Сбербанк ао", "raw_name": "Сбербанк ао"}
   ```
   `raw_name` сохраняется ради аудита: показать, что было в исходной картинке. `ticker` резолвится через `data/tickers.json` aliases.
3. **Отображение** на GitHub Pages — отдельная страница `mages_index.html`: таблица текущего и исторических составов, веса, изменения месяц-к-месяцу.
4. **Comparison-фича.** Несколько вариантов наложения, выбираемых пользователем:
   - Q1 ∩ Mages Index (только акции, попавшие в оба) — построить equity curve этого «consensus» портфеля и сравнить с чистым Q1 и чистым Mages Index.
   - Mages Index веса × Q1 фильтр (берём акции из Mages, но только те, что в Q1; ребалансировка по весам Mages Index с пропорциональным re-weight).
   - Q1 веса × Mages Index фильтр (зеркальная версия).

## Что нужно от основного pipeline

Закладывается в `task 000`:

- **Ticker dictionary с alias-резолвом** (phase 3) уже умеет переводить «Сбербанк ао» → SBER. Без этого парсинг картинок не сходится с holdings.
- **Backtest как чистая функция** от signal и universe-fn (phase 9) — это позволяет легко добавить `MagesIndexSignal` (signal равен 0 для всех, кроме тикеров из Mages, для которых = вес × произвольная константа) и пробросить его через ту же `backtest()` функцию.

## Claude skill: `/parse_mages_index`

Подход: юзер кладёт картинку в `raw_sources/`, дёргает `/parse_mages_index <path>`. Skill:
1. Читает картинку (multimodal).
2. Парсит таблицу `доля% / название`.
3. Резолвит названия в тикеры через `data/tickers.json`. Неоднозначности (кириллица, варианты написания) — спрашивает пользователя.
4. Создаёт `data/mages_index/YYYY-MM.jsonl`.
5. Если в индексе есть инструменты, не являющиеся акциями TQBR (ОФЗ, корп. облиги, валюта USD из апрельского списка) — записывает их с `type=ofz/bond/fx`, не пропускает; эти записи не попадут в Q1-сравнение, но останутся в исходном списке индекса.

Поэтому `data/tickers.json` нужно расширить: добавить записи для ОФЗ (типа `OFZ_26238`) и облигаций. На основном плане они только декларируются, без ингеста (потому что момент-pipeline ими не пользуется).

## Ключевые вопросы для будущей итерации

- **Точность OCR vs ручной аппрув.** Картинки с округлёнными процентами (16,53%) — OCR ошибается на запятых/точках. Skill должен показывать diff и требовать аппрув.
- **Ребаланс индекса магов.** Картинки приходят с какой частотой — раз в месяц? Чаще? Для построения equity curve нужно знать дату каждого состава (вытащить из имени файла или из контекста, который юзер укажет в команде).
- **Как сравнивать с Q1, если универс survivorship-free?** Q1 каждый месяц считается на доступных данных. Mages Index — на subjective выборке автора, без формального универса. Простой пересечение (Q1 ∩ Mages) теряет много имён. Альтернатива — сравнивать Mages Index с **Q1+Q2** (топ-50% импульса).
- **Веса индекса магов с фрагментами** (`0,21%`, `0,42%`) — индикаторы микропозиций. Влияют ли они реально на performance? Если нет — фильтровать ниже какого-то порога? Это вопрос для исследования.
- **Облигации в Mages Index.** Они имеют другую природу доходности (карри + дюрация). Их совместное рассмотрение с акциями в одной equity curve — методически некорректно, но именно так автор делает. Решение: показывать total Mages Index (как у автора) ОТДЕЛЬНО от Mages-Equity (только акции), сравнение с Q1 — только Mages-Equity.
