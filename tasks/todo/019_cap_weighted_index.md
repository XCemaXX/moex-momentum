# Cap-weighted index: воспроизведение MCFTRR + альтернативные веса

> Создана 2026-05-20. Контекст собран в research-сессии (endpoints проверены live curl 2026-05-20). Подробный staged-план не пишем заранее — решаем по ходу. Это файл-контекст, чтобы не терять найденное.

## Цель

Пользователь хочет «лучше взвешенный» индекс российских акций и сравнение схем взвешивания. Три шага + опциональный четвёртый:

1. **Воспроизвести индекс** (IMOEX → MCFTRR) собственным движком — валидация, что машинка считает правильно.
2. **Full-cap версия**: тот же состав, но вес строго по полной капитализации (`price × shares`), без free-float и без капов.
3. **Сравнить** 1 vs 2.
4. **(later)** Взвесить собственную Q1 по капитализации и сравнить с equal-weight Q1.

Смысл «reproduce → reweight на одном движке»: единственная меняющаяся переменная — схема весов, расхождение чисто атрибутируется ей, а не разнице методологий.

## Важно: корректировка постановки (locked)

Пользователь изначально считал IMOEX «плохо взвешенным», т.к. в таблице smart-lab вес ≠ капитализация. Это **не дефект** — IMOEX/MCFTRR взвешен по **free-float** капитализации с капом на эмитента (исторически ~15%). Расхождение вес/капа = именно free-float + capping.

Доказательство (snapshot smart-lab IMOEX, 2026-05-20):

| Компания | Полная капа, ₽млрд | Ранг по капе | Вес | Ранг по весу |
|---|---|---|---|---|
| Роснефть | 4 298 | #2 | 2.73% | #12 |
| Новатэк | 3 408 | #4 | 4.12% | #7 |
| Полюс | 2 835 | #6 | 3.54% | #10 |
| ЛУКОЙЛ | 3 568 | #3 | **15.72%** | **#1** |

Причина — free-float: у Роснефти/Новатэка/Полюса контролирующий собственник (государство, Михельсон/Тимченко, Керимов), free-float мал → малый вес, хотя капа огромная. У ЛУКОЙЛа высокий float → большой вес и упор в кап-лимит.

**Следствие.** Full-cap (то, что хочет пользователь) — это НЕ «улучшение», а наивная схема: усиливает концентрацию в неторгуемых госпакетах. Пользователь это принял и хочет именно её — как исследовательский эксперимент, не как «правильный» индекс. Если реальная цель окажется «снизить концентрацию» — full-cap её ухудшает, тогда смотреть equal-weight / capped-weight (это уже другой разговор).

## Что MOEX реально отдаёт (verified 2026-05-20)

Все запросы: base `https://iss.moex.com/iss`, `iss.meta=off`. **`.json` в пути обязателен** (без него — XML). В sandbox у ISS self-signed cert (MITM-прокси) — качать с `dangerouslyDisableSandbox` либо через проектный httpx-стек, который ходит нормально.

### ✅ Состав индекса + официальные веса (для шага 1)

```
GET /iss/statistics/engines/stock/markets/index/analytics/{INDEXID}.json
    ?date=YYYY-MM-DD&iss.meta=off
```

- `{INDEXID}` в **пути** (например `IMOEX`). Форма `analytics.json?index=IMOEX` отдаёт лишь СПИСОК индексов, не состав.
- Блок `analytics`, колонки: `indexid, tradedate, ticker, shortnames, secids, weight, tradingsession, trade_session_date`.
- `weight` — официальный вес в % (free-float + капы уже зашиты).
- История назад до **2001-01-03** (блок `analytics.dates`).
- Пагинация: `analytics.cursor` (PAGESIZE=20), плюс `PREV_DATE`/`NEXT_DATE` для шага по торговым дням.
- Это всё, что нужно для воспроизведения официального индекса.

### ⚠️ Число акций / капитализация — ТОЛЬКО текущий снимок

```
GET /iss/securities/{TICKER}.json?iss.only=description   → ISSUESIZE (объём выпуска)
GET /iss/engines/stock/markets/shares/boards/TQBR/securities/{TICKER}.json
    → securities.ISSUESIZE, marketdata.ISSUECAPITALIZATION (полная капа, real-time)
```

Пример SBER: `ISSUESIZE=21586948000`, `ISSUECAPITALIZATION=6983161808520`. Это **point-in-time**, исторического ряда нет.

### ❌ Историческое число акций / историческая капа — НЕТ

Daily history endpoint (`/iss/history/.../shares/.../{TICKER}.json`) колонок cap/issuesize **не содержит** (проверено: только `VOLUME`). MOEX исторический share-count ряд не публикует.

## Блокер шага 2 и принятый компромисс

Full cap = `price × shares`. На сегодня — точно. В прошлом — числа акций нет, реконструируем приближённо:

> `shares_today` back-adjusted по сплитам (сплиты у нас уже есть в `data/splits/`).

Ловит сплиты, **не ловит**: buyback'и (LUKOIL гасит → в прошлом акций было больше, занизим капу) и допэмиссии (VTB, банки, T → в прошлом меньше, завысим). На уровне индекса размывается, на отдельных именах ошибка реальна.

## Движок (эскиз, детали по ходу)

Один reusable reconstructor: вход — состав + вектор весов + дневные доходности; выход — ряд индекса.

- `index_return(t) = Σ wᵢ(t) · rᵢ(t)`.
- `rᵢ` — дневная доходность из наших цен (split-adjusted). Для **price index** (IMOEX) — без дивидендов. Для **total return** (MCFTRR) — + реинвест дивидендов нетто 13% (налог уже в `config.py`, `DIVIDEND_TAX`).
- Режим **reproduce**: `wᵢ` = официальный вес из analytics.
- Режим **full-cap**: `wᵢ = capᵢ / Σcap`, `capᵢ = priceᵢ × sharesᵢ(reconstructed)`, состав — тот же официальный из analytics.
- Сравнивать ряды доходностей, чтобы обойти дивизор/базовый уровень (точное воспроизведение уровня индекса нереально и не нужно).

**Планка валидации шага 1:** не bit-exact. Tracking накопленной доходности vs published MCFTRR в пределах ~1%, корреляция дневных ретёрнов >0.99. Стать новым regression-якорем рядом с VSMO=4.6458%. Рекомендация: сначала воспроизвести IMOEX (price, без дивов — меньше движущихся частей), потом добавить дивиденды → MCFTRR.
