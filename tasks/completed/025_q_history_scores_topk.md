# Скоры в q_history: сортировка Q1–Q4, номер строки, выбор top-K в «Изменениях»

> Страница «Состав Q1–Q4 по итогам месяца» (`q_history.html`). Четыре связанные
> фичи, у всех одна зависимость — отсортировать Q1–Q4 по momentum-скору на сборке.
> Связь: `task 024` (вывод: держать top-15–20 из top-100 в выборке не хуже).

## Проблема / текущее состояние

- Холдинги на диске (`data/momentum/<signal>/holdings/YYYY-MM.json`) хранят только
  списки тикеров, отсортированные **по алфавиту** (`quartile_split` → `sorted(...)`).
- Скор `(0.9·r(12-1)+0.1·r(6-1))/σ(12)` в `backtest` считается (`signal.compute`),
  но **не сохраняется** — `quartile_split` его отбрасывает.
- Значит на странице Q1–Q4 идут по алфавиту, а не по силе сигнала, и «Изменения Q1»
  сравнивают полные Q1-множества (25 имён).

## Зависимость (крючок всех фич)

Сохранить скор по тикеру за месяц при бэктесте (он уже считается — донести из цикла
`backtest` через `BacktestResult` в `write_backtest`) в `scores.csv` на диск. **Скор
используется только для предсортировки `data.json` на сборке; само число в `data.json`
НЕ кладём** — порядок массива и есть ранг. Тай-брейк — **score DESC, ticker ASC** (как в
`quartile_split`/`score_ranking`), чтобы top-K на странице совпадал с top-K из task 024.

## Фичи (все — клиентский рендер, бэкенд только сортирует на сборке)

### A. Сортировка Q1–Q4 по q-фактору
`data.json` приходит уже отсортированным по скору DESC; `renderQGrid` рендерит в этом
порядке. Сам скор не показываем.

### B. Порядковый номер строки
Колонка `#` (1..N) во всех квартилях = ранг по q-фактору (= индекс в пришедшем массиве).
Ранг сохраняется при сортировке по имени.

### C. Селектор top-K в «Изменениях Q1»
- `renderDiff` берёт top-K по скору (= первые K пришедшего массива) из Q1 текущего и
  предыдущего месяца и диффит эти подмножества.
- Контрол: number-input + пресеты **10/15/20** + «Полный Q1»; дефолт — полный.
  (25 не нужен — полный Q1 = 25 при top-100.)
  Произвольное K — бесплатно (JS режет первые K). K клампится к |Q1|.
- Заголовок «Изменения Q1» → «Изменения top-K» (динамически).
- Reason-строки учитывают границу top-K: «был в Q1 (#ранг)» / «остался в Q1 (#ранг)»
  для имён, что в Q1, но вне среза; иначе как раньше (был в Qx / новичок / ушёл / делист).
- Семантика: «как если бы держал top-K». Прямой инструмент к выводу task 024.

### D. Тумблер сортировки имя / q-фактор
Кнопки «по q-фактору» (дефолт) и «по имени» над сеткой Q1–Q4 — чтобы было явно видно,
как отсортированы столбики. «по q-фактору» = порядок из `data.json`; «по имени» =
переставить по тикеру (ранг в колонке `#` сохраняется).

## Зафиксировано

- **Скор не показываем и не шлём в `data.json`** — клиенту достаточно порядка массива.
  Полная точность скора остаётся в `scores.csv` на диске; если позже захотим показывать
  число — это однострочное добавление в `_build_data_json` без работы по бэкенду.
- **Сортируем все Q1–Q4** по убыванию скора (не только Q1). Номер строки — во всех.

### Хранение скора — выбрано: scores.csv на диске, в data.json только порядок

- `backtest` доносит `scores` через `BacktestResult` → `write_backtest` персистит
  `scores.csv` (`month,ticker,score`, полная точность). Holdings-файлы не трогаем.
- `site_builder._build_data_json` читает `scores.csv`, **предсортирует Q1–Q4 по ПОЛНОМУ
  скору** и кладёт строки **двухэлементными** `[ticker, canonical]` в этом порядке.
- Значение скора по проводу не идёт → data.json не растёт, и ранг невозможно случайно
  пересчитать по округлённому полю (его просто нет).

Отвергнуто: (Вариант 2) округлённый скор третьим элементом — даёт +23% к data.json и
соблазн пересортировать по округлению (ломает границу top-K vs task 024); не нужен, пока
скор не показываем. (3) холдинги в порядке скора — ломает алфавит holdings-файлов и
семантику `quartile_split`.

### ИНВАРИАНТ (зафиксирован в коде/комментарии и в methodology)

**Порядок массивов Q1–Q4 в `data.json` = авторитетный ранг по ПОЛНОМУ скору**
(DESC, ties → ticker ASC). Число скора в `data.json` отсутствует намеренно.

Следствия:
- «по q-фактору» в UI = **рендер исходного порядка из data.json** (как пришло).
- «по имени» = клиент пересобирает по тикеру; ранг (`#`) переносится с каждой строкой.
- top-K и номер строки всегда берутся из исходного порядка массива.

## Acceptance

1. Скор персистится при бэктесте в `scores.csv`; `data.json` отдаёт Q1–Q4
   предсортированными по полному скору, строки двухэлементные (обратно совместимо).
2. Все Q1–Q4 в порядке скора DESC, ties → ticker ASC; есть колонка `#` = ранг.
3. «Изменения» считаются для выбранного K; пресеты 10/15/20 + «Полный Q1»; K≤|Q1|.
4. Срез top-K совпадает с ранжированием из task 024 (тот же тай-брейк).
5. Старые потребители data.json (диффы, pending, canonical lookup) не сломаны.
6. Тумблер имя/q-фактор переключает порядок сетки, ранг в `#` сохраняется.
7. Тесты на сборку data.json (порядок по скору) + рендер; ruff/mypy чисто.
8. Инвариант порядка задокументирован (комментарий в `_build_data_json` + methodology).

## Не входит

- Пересчёт NAV/доходности top-K на странице (это уже есть в `compare.html` веере
  концентрации, task 024). Здесь — только состав и его изменения.
- Показ числового значения скора (намеренно).
