# Site explorer: интерактивный мульти-серийный график (механизм)

> Пре-задача для `task 21`. Делается одновременно с `task 9` — первый контент explorer'а = серии из task 9 (`q1_simple`, `q1_curve_fit`, MCFTRR), и это же первый тест механизма.

## Цель

Новая отдельная страница сайта `docs/pages/compare.html` с **одним** plotly-графиком, где пользователь сам тоглит, какие именованные линии показывать. Механизм должен расширяться: добавление новой стратегии = регистрация новой серии, без правок кода графика. Текущие страницы (curve_fit-only) не трогаем.

## Решения (locked с пользователем)

- **Explorer показывает оба сигнала** (`simple` и `curve_fit`) рядом. Это снимает запрет task 9 «site остаётся curve_fit-only» **только для этой страницы** — основные/дефолтные страницы и default `--signal` остаются `curve_fit`. Статистический research-док task 9 живёт отдельно.
- **Гранулярность линий: Q1 + бенчмарки, по стратегиям.** Каждая стратегия даёт 1-2 кривые (напр. `q1_simple`, `q1_curve_fit`, MCFTRR). НЕ полные Q1-Q4 на каждую стратегию — это перегрузит UI. Полный Q1-Q4 остаётся на существующих per-signal страницах.
- **Отдельная страница**, добавляется в навигацию рядом с текущими.

## Архитектура (эскиз, детали по ходу)

**Series registry.** Список именованных серий; каждая:

```
id, label, nav: pd.Series[Period[M]], kind, default_visible, dims: dict[str, str]
```

- `nav` — кумулятивная кривая (старт нормируется ребейзом, не хардкодом).
- `kind` — `"strategy"` | `"benchmark"`. Отделяет Q1-комбинации от бенчмарков (MCFTRR и т.п.); каскад/фильтр UI применяется только к `strategy`.
- `dims` — **открытый** словарь осей-тегов, НЕ фиксированные поля. Q1-стратегия: `{"quartile":"Q1","formula":"curve_fit","weighting":"equal","filter":"none"}`. Бенчмарк: `dims={}`.

Почему открытый dict, а не фиксированные `(formula, weighting, filter)`: бенчмарки не лезут в три поля; оси сами вырастут (rebalance, stop-loss, lookback, params); новая ось = новый ключ без миграции существующих серий. Цена та же, что у фиксированных полей. Каскад (отложен) итерирует по объединению ключей `dims` среди `kind="strategy"` и строит по дропдауну на ключ — работает для 3 осей и для 6 без правок кода.

Реестр собирается в `src/viz/` функцией-провайдером; на старте провайдеры берут серии из `data/momentum/{signal}/q_values.csv`. Добавить стратегию = добавить провайдер, отдающий серии с заполненным `dims`.

**Не переусложнять сейчас:** `dims: dict[str,str]` + `kind` — весь задел. Без типизированной схемы/валидации/онтологии осей (YAGNI). Консистентность значений осей (`"curve_fit"`, не `"cf"`) — забота `task 21`, когда комбинаций станет много.

**График.** Одна plotly-фигура, каждая серия = trace. Тоггл — нативный клик по легенде plotly (click-to-hide, double-click-isolate) — это уже «выбор нескольких линий». Переиспользовать `_rebase_buttons` из `src/viz/plotly_charts.py`.

**Ребейз обязателен.** Серии стартуют с разных месяцев (`simple`/`curve_fit` с 2012-12; mages/cap-weighted из task 2/19 — позже). График должен корректно жить с рваными стартами, а ребейз нормирует от выбранной общей даты. Без этого сравнение визуально врёт.

**Статика, без сервера.** Страница — pre-built артефакт (`serve_site` раздаёт только готовое). Все серии **прекомпьютятся на build-time** и вшиваются в HTML; тогглинг — клиентский поверх готовых traces. Живого пересчёта нет.
